# Mock Data Summary

## Introduction

This document provides a consolidated, module-wise overview of all mock datasets used by the dashboard frontend. It explains what each dataset represents, where it is used in the UI, the data format and key fields, and includes realistic sample data snippets sourced directly from the codebase. The mock data is designed to support the dashboard’s filters (time preset, tenant, department, team, user, feature/module, geography) and to keep widgets working consistently until APIs are wired.

## Preserved Prose Summary and Context

The dashboard frontend uses mocked data to power widgets across all sections while keeping interfaces ready for API integration. The mocks are intentionally rich:

- Programmatic coverage: The generator spans daily, weekly, and monthly periods, and tags records with department, team, feature/module, and region to ensure filters in the Global Toolbar always yield visible data.
- UI fidelity: Datasets are structured to match chart and table expectations. For instance, the Adoption & Engagement section draws DAU/WAU/MAU from a unified series (mockTimeSeries), module usage tables from mockModuleUsage, department growth trends from mockDepartments, and funnels from mockDiscoveryFunnel. Advanced visuals like time heatmaps and network diagrams have dedicated structures (mockTimeHeatmap, mockNetwork).
- Role-based previews: Some widgets include role badges and preview scaffolding in the UI. While RBAC is placeholder-only in the UI at present, mock datasets are still annotated with logical tags useful when server-side RBAC and data scoping are introduced.
- Drill-down friendly: Most cards provide tabular data sources that can be exported via PNG and, in some cases, CSV logic. The tables render values directly from the mocks and respond to filters for search, sorting, and scoping.
- Non-functional behavior: The mock datasets support exporting (PNG/PDF) without additional libraries, and the structure of the data is designed to be easily replaced by live API responses later, preserving the same shape and keys.

Design guidelines and coverage include:
- Clean, responsive, themeable components built with Tailwind CSS and lightweight visualization primitives.
- Consistent legends, labels, and units; glossary tooltips for metric definitions.
- End-to-end drill-down flow from KPI to visualization to tables/logs.
- Key sections: Adoption & Engagement; Effectiveness; Training & Awareness; Organizational Metrics; Feedback & Problem Detection; Team Analytics; Usage Patterns; Feature Flag & Rollout Tracking; Cost & Credits; and Kavia-Admin Only analytics.
- Tables, logs, and analytics include user/team/feature rankings, onboarding milestones, support trends, workflow funnels, cost/credits utilization, and admin-specific performance/compliance metrics.

As APIs are integrated, each dataset can be replaced by real endpoints that return the same structures:
- Time series: { label, period, …metrics…, team?, dept?, region? }
- Tables: consistent, typed rows keyed by the columns used in the UI
- Heatmaps: matrices with x/y labels
- Graph/network: nodes and links with stable identifiers
- Funnels and allocations: clearly named fields for each stage or metric

This approach allows the dashboard to remain usable and testable, with mock data providing a realistic experience while backend services are under development.
